## 1. 文档说明

### 1.1 设计原则

1. **规范化与反规范化结合**：核心业务表遵循第三范式（3NF），减少数据冗余；高频查询表适当反规范化，提升查询性能
2. **统一命名规范**：表名采用`{业务域}_{实体名}`格式，全部小写，下划线分隔；字段名采用`{属性名}`格式，全部小写，下划线分隔
3. **统一审计字段**：所有业务表包含`created_by`、`created_at`、`updated_by`、`updated_at`、`deleted`、`version`六个审计字段
4. **软删除策略**：所有业务表采用软删除（`deleted`字段），不物理删除数据
5. **乐观锁控制**：所有业务表包含`version`字段，支持乐观锁并发控制
6. **主键策略**：采用雪花算法（Snowflake）生成分布式唯一ID，bigint类型

### 1.2 命名规范

| 对象类型 | 命名规则 | 示例 |
|---------|---------|------|
| 数据库 | 项目名_sn | pub_supervision |
| 表 | 业务域_实体名（复数） | identification_tasks |
| 字段 | 属性名（小写下划线） | publication_isbn |
| 主键 | id | id |
| 外键 | 关联表名_主键名 | publication_id |
| 索引 | idx_表名_字段名 | idx_tasks_isbn |
| 唯一索引 | uk_表名_字段名 | uk_users_username |
| 外键约束 | fk_表名_关联表名 | fk_tasks_publications |

### 1.3 数据类型规范

| 数据类型 | 适用场景 | 说明 |
|---------|---------|------|
| BIGINT | 主键、外键、计数 | 雪花算法ID |
| VARCHAR(n) | 短文本 | n为字符数 |
| TEXT | 长文本 | 不定长文本 |
| LONGTEXT | 超长文本 | 文档内容、JSON数据 |
| DECIMAL(p,s) | 金额、评分 | p总位数，s小数位 |
| INT | 整数 | 状态、数量 |
| TINYINT | 枚举值 | 0-255范围 |
| DATETIME | 日期时间 | 精确到秒 |
| TIMESTAMP | 时间戳 | 自动更新 |
| JSON | 结构化数据 | 复杂嵌套数据 |
| BLOB | 二进制数据 | 文件内容（建议存对象存储） |

**建表SQL：**

```sql
CREATE TABLE `sys_user` (
    `id` BIGINT NOT NULL COMMENT '主键ID',
    `username` VARCHAR(50) NOT NULL COMMENT '用户名',
    `password` VARCHAR(100) NOT NULL COMMENT '密码（BCrypt加密）',
    `real_name` VARCHAR(50) NOT NULL COMMENT '真实姓名',
    `nickname` VARCHAR(50) DEFAULT NULL COMMENT '昵称',
    `gender` TINYINT DEFAULT 0 COMMENT '性别：0未知 1男 2女',
    `email` VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
    `phone` VARCHAR(20) DEFAULT NULL COMMENT '手机号',
    `avatar` VARCHAR(255) DEFAULT NULL COMMENT '头像URL',
    `id_card` VARCHAR(255) DEFAULT NULL COMMENT '身份证号（AES加密）',
    `department_id` BIGINT DEFAULT NULL COMMENT '部门ID',
    `position` VARCHAR(50) DEFAULT NULL COMMENT '职位',
    `employee_no` VARCHAR(50) DEFAULT NULL COMMENT '工号',
    `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0禁用 1启用 2锁定',
    `last_login_at` DATETIME DEFAULT NULL COMMENT '最后登录时间',
    `last_login_ip` VARCHAR(50) DEFAULT NULL COMMENT '最后登录IP',
    `login_count` INT NOT NULL DEFAULT 0 COMMENT '登录次数',
    `password_updated_at` DATETIME DEFAULT NULL COMMENT '密码更新时间',
    `created_by` BIGINT DEFAULT NULL COMMENT '创建人',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `updated_by` BIGINT DEFAULT NULL COMMENT '更新人',
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `deleted` TINYINT NOT NULL DEFAULT 0 COMMENT '是否删除：0否 1是',
    `version` INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_users_username` (`username`),
    UNIQUE KEY `uk_users_email` (`email`),
    KEY `idx_users_phone` (`phone`),
    KEY `idx_users_dept` (`department_id`),
    KEY `idx_users_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

## 9. 索引设计

### 9.1 索引设计原则

1. **主键索引**：所有表必须有主键索引，采用BIGINT类型
2. **唯一索引**：业务唯一字段必须建立唯一索引（如用户名、ISBN、任务编号）
3. **外键索引**：所有外键字段必须建立普通索引
4. **查询索引**：高频查询条件字段建立复合索引，遵循最左前缀原则
5. **排序索引**：高频排序字段建立索引
6. **全文索引**：文本搜索字段建立全文索引（如书名、舆情内容）

### 9.2 复合索引设计

| 表名 | 索引名 | 字段 | 适用场景 |
|------|--------|------|---------|
| idt_identification_tasks | idx_task_status_assignee | (status, assignee_id) | 按状态和鉴定人查询任务 |
| circ_packages | idx_package_status_shipped | (status, shipped_at) | 按状态和时间查询包裹 |
| circ_alerts | idx_alert_level_status | (alert_level, status) | 按等级和状态查询预警 |
| opi_opinions | idx_opinion_channel_publish | (channel_id, publish_time) | 按渠道和时间查询舆情 |
| opi_opinions | idx_opinion_sentiment_risk | (sentiment, risk_level) | 按情感和风险查询舆情 |

### 9.3 索引维护

1. **定期分析**：每月执行`ANALYZE TABLE`更新索引统计信息
2. **慢查询优化**：每周分析慢查询日志，优化索引
3. **索引碎片整理**：每季度对高频更新表执行`OPTIMIZE TABLE`
4. **冗余索引清理**：每月检查并清理冗余索引
